# VQA Codec Testing

## Why this exists

The VQA bug cluster (TIM-587, TIM-602, TIM-604, TIM-613) cost roughly 3 days of
debugging.  Quantitative metrics (fill%, frame count) passed while frames showed
block-aligned corruption.  A pixel-diff harness would have caught each regression
within seconds.

## How it works

`scripts/vqa-pixel-diff.py` compares our Python reference decoder
(`scripts/vqa_decode_verify.py`) frame-by-frame against **ffmpeg's independent
VQA decoder**.  If the p99 per-pixel-channel delta exceeds a configurable
threshold (default: 5), the harness exits 1 (FAIL).

The comparison happens at native VQA resolution (no blit scaling) so blit
geometry bugs don't add noise.

## Quick start

```bash
# Test on the committed synthetic VQA (no game data needed):
python3 scripts/vqa-pixel-diff.py e2e/goldens/vqa/test.vqa --frames 0,1,2

# Test on real game VQA files (requires MAIN.MIX from game install):
python3 scripts/vqa-pixel-diff.py /path/to/MAIN.MIX

# Or point at a pre-extracted VQA directly:
python3 scripts/vqa-pixel-diff.py /tmp/extracted_000.vqa --frames 0,29,59
```

Exit codes: **0** = pass, **1** = fail, **2** = skip (data or ffmpeg absent).

## CI integration

The `vqa-pixel-diff` job in `.github/workflows/ci.yml` runs two checks on every
PR:

| Step | Gate | Skips when |
|------|------|-----------|
| Synthetic VQA | Always runs | Never |
| Game MAIN.MIX | Runs if `build/run-172/MAIN.MIX` present | Data absent |

The synthetic VQA (`e2e/goldens/vqa/test.vqa`) is committed to the repo and is
regenerated from `scripts/gen_test_vqa.py` — the CI job also checks that the
committed file matches the generator output exactly.

## Adding golden frames for a new VQA

```bash
python3 scripts/vqa-pixel-diff.py /path/to/file.vqa \
    --generate-goldens \
    --goldens-dir e2e/goldens/vqa/
```

Golden frames are written to `e2e/goldens/vqa/<stem>/golden_NNNN.png`.  They
are derived from game assets and **must not be committed** to the repo.

## The synthetic test VQA

`e2e/goldens/vqa/test.vqa` is a 2,640-byte synthetic file generated by
`scripts/gen_test_vqa.py`.  It is safe to commit (our own test data, no game
assets).  It exercises:

- **CBFZ** full-codebook decode (4-entry, 4×4 blocks)
- **VPTZ** compressed frame-pointer array (regular codebook lookup)
- **Solid-colour markers** `hi=0xFF` for blockH=4 (TIM-587 regression class)
- **CBPZ** single-part partial codebook; new codebook active on next frame
- **6-bit VGA palette** scaling (CPL0 → 8-bit RGB)

To regenerate `test.vqa` after changing the generator:

```bash
python3 scripts/gen_test_vqa.py e2e/goldens/vqa/test.vqa
git add e2e/goldens/vqa/test.vqa
```

## Threshold guidance

The default threshold of **p99 ≤ 5** (per pixel-channel delta) is chosen so:

| Scenario | Typical p99 |
|----------|-------------|
| Correct decoder vs ffmpeg | 0–2 |
| Palette off-by-one | 2–4 |
| Wrong codebook index (wrong block colour) | 50–200 |
| Black-square bug (TIM-587/613 class) | 50–255 |
| Missing audio (not visible) | 0 |

Raise to 10 if you see consistent failures around 6–8 (likely minor VGA palette
rounding differences for a specific file).  Never raise above 20 without a clear
diagnosis — that range hides real visual corruption.

## The Python reference decoder

`scripts/vqa_decode_verify.py` is a Python port of `linux/win32-stubs/vqa_player.cpp`.
As of TIM-639 it correctly implements:

- 6-bit VGA palette expansion `(v << 2) | (v >> 4)` — matches C++ and ffmpeg
- Solid-colour codebook pre-fill for both blockH=4 (0xFF00–0xFFFF) and
  blockH=2 (0x0F00–0x0FFF) ranges (TIM-587 / TIM-613)
- CBFZ / CBPZ LCW decompression and partial-codebook accumulation
- VPTZ / VPRZ decompressed frame-pointer arrays

If `vqa_player.cpp` changes in a way that affects frame output, update
`vqa_decode_verify.py` to match and re-run the harness to confirm p99 stays ≤ 5.
