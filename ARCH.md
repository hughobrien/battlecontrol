# Linux Port Architecture — TiberiaDawn and RedAlert

## Overview

Command & Conquer: Tiberian Dawn (TD) and Red Alert (RA) were originally written
for MS-DOS (TD) and Windows 95 / Win32 (RA) using Westwood's proprietary WIN32LIB
graphics and audio layer, DirectDraw, DirectSound, and Win32 CRT extensions. This
repo ports both games to build and run as native Linux binaries using GCC/Clang,
CMake, and SDL2.

The goal is a native Linux executable that can be built from source and run against
the original game data from the Steam CnC Remastered Collection.

## Why Not Wine?

Wine was considered and ruled out in favour of a native port. The reasons:

- **Open source build chain.** A native port compiles with GCC/Clang/CMake and
  links only against SDL2 — no proprietary runtime or compatibility
  layer required. The result can be built and run on any Linux system without a
  Wine installation.

- **Debuggability.** Wine abstracts the Win32 layer in ways that make crashes
  and memory bugs harder to trace. A native binary runs under ASAN, Valgrind,
  and GDB directly without Wine interposing.

- **Control over the platform surface.** The games' real OS dependencies turned
  out to be narrow: graphics (DirectDraw), audio (DirectSound), input (Win32
  messages), and file I/O. Replacing each with a thin SDL2 shim gave precise
  control and predictable behaviour, whereas Wine would silently re-implement
  the same Win32 semantics with its own quirks.

- **CI compatibility.** Native binaries run headlessly under Xvfb in a standard
  Linux CI environment without needing a Wine prefix, display driver workarounds,
  or 32-bit multilib dependencies.

The two references to Wine in the codebase (`REDALERT/WIN32LIB/DDRAW.CPP` and
`scripts/first-compile-pass41F.sh`) are design-inspiration comments only —
they note that the SDL2 input event-filtering approach mirrors how Wine's
winex11 driver isolates window-class events. Wine itself is not used.

## Build System

The build is driven by `CMakeLists.txt` at the repo root using Ninja.

Key targets:

| Target | Sources |
|--------|---------|
| `ra` | `REDALERT/*.CPP`, `REDALERT/WIN32LIB/*.CPP`, Linux stubs |
| `td` | `TIBERIANDAWN/*.CPP`, Linux stubs (not WIN32LIB) |

Both targets share:

- `-DLINUX=1 -DUNIX=1 -DPOSIX=1` compile definitions
- A force-include of `linux/win32-stubs/msvc-compat.h` (provides `_stricmp`,
  `_strdup`, `strupr`, and other MSVC CRT shims)
- `td` additionally requires `-DWIN32=1` because `TIBERIANDAWN/FUNCTION.H` has
  the `WIN32` define commented out; without it `RawFileClass::Handle` is typed
  inconsistently across translation units, causing LP64 ODR crashes

Case-folded include shims are generated by `scripts/generate-include-shim.py`
into `build/include-shim/{redalert,tiberiandawn,td-win32lib}/`. These symlink
upper-case source headers (e.g. `FUNCTION.H`) to lower-case names so that
`#include "function.h"` works on a case-sensitive filesystem.

## Platform Abstraction Layer

Win32 and DOS APIs are replaced with thin stubs and SDL2 equivalents. The stubs
live under `linux/win32-stubs/`.

### Graphics — DirectDraw replaced with SDL2

`REDALERT/WIN32LIB/DDRAW.CPP` is the primary RA graphics layer. The original
DirectDraw path is replaced by an SDL2 path gated on `!defined(_MSC_VER)`:

- `SDL_Window` + `SDL_Renderer` + 8-bit indexed `SDL_Surface` replace the
  `LPDIRECTDRAWSURFACE` primary
- Palette updates use `SDL_SetPaletteColors` (the `Set_DD_Palette` call site)
- `Wait_Vert_Blank` calls `SDL_Process_Input_Events` (see Input below) and
  drives `SDL_RenderPresent` once per frame
- `SDL_WINDOWEVENT_FOCUS_*` feeds `GameInFocus` so `Map.Render()` only fires
  when the window has focus

TD uses `TIBERIANDAWN/WIN32LIB/DDRAW.CPP` (the original stub-only version) plus
`linux/win32-stubs/sdl_window.cpp` for window lifecycle.

### Audio — DirectSound replaced with SDL2 mixer

`REDALERT/AUDIO.CPP` has a non-`_MSC_VER` block that implements device open,
mixing, ADPCM streaming, and track scheduling using SDL2's audio subsystem
(`SDL_OpenAudioDevice`, `SDL_QueueAudio`). The Windows MMSYSTEM/DirectSound path
is elided by the `#ifndef _MSC_VER` guard.

### Input — SDL keyboard and mouse

`REDALERT/KEY.CPP` implements `SDL_Process_Input_Events()`, called once per
frame from `Wait_Vert_Blank`. It drains `SDL_KEYDOWN`/`SDL_KEYUP` events,
maps SDL keysyms to Westwood `VK_*` codes, packs `WWKEY_*BIT` modifier bits,
and calls `_Kbd->Put()`. Mouse button and motion events update `SDL_Cursor_X/Y`
(read by `GetCursorPos` stubs).

TD has SDL keyboard wired via the same `SDL_Process_Input_Events` pump in
`linux/td-win32-stubs.cpp` (TIM-383 + TIM-856). `WWKeyboardClass::Check()`
calls `Wait_Vert_Blank()` → `SDL_Process_Input_Events()` on every poll, so
SDL_KEYDOWN/KEYUP events reach the engine keyboard buffer through all code
paths including the main menu gadget system. Frame-counter cheat injection
(`TD_CHEAT=1`) is kept for automated smoke testing.

### Win32 CRT shims

`linux/win32-stubs/msvc-compat.h` provides:
- `_stricmp` / `_strnicmp` → `strcasecmp` / `strncasecmp`
- `_strdup` → `strdup`
- `strupr` / `strlwr` — in-place ASCII case converters
- `_splitpath` — POSIX reimplementation

### File I/O

`linux/win32-stubs/posix_fileio.cpp` implements `CreateFileA`, `CloseHandle`,
`ReadFile`, `WriteFile`, `SetFilePointer`, and `GetFileSize` over POSIX `open`/
`read`/`lseek`. This allows `RawFileClass` (used by both games for `.MIX` asset
loading) to work without Win32.

### DDE stubs

`TIBERIANDAWN/CCDDE.CPP` compiles with `WIN32=1` active, providing
`DDEServerClass` no-ops. `Instance_Class` stubs (Win32 DDE client, unused on
Linux) live in `linux/td-win32-stubs.cpp`.

## Graphics Pipeline

Both games target a 640x480 viewport:

1. `STARTUP.CPP` (both games) calls `Read_Setup_Options`, which originally
   writes the Remastered internal resolution (1536x3072). A post-call override
   forces `ScreenWidth=640, ScreenHeight=480`.
2. `DDRAW.CPP::Set_Video_Mode` creates an `SDL_Window` (640x480, shown) and
   an 8-bit indexed `SDL_Surface` of the same size.
3. Each frame: `HidPage` (system-memory surface) is blitted to the SDL primary
   surface, then presented via `SDL_RenderPresent`.
4. Under Xvfb the window renders off-screen; the `scripts/run-td-cheat.sh` and
   `scripts/setup-run-ra-remastered.sh` harnesses start Xvfb on `:99`.

## Mix File / Asset Loading

`.MIX` archives are located by `MixFileClass::Cache` using the POSIX file I/O
shims above. On Linux the working directory must contain the mix files; the
setup scripts (`scripts/setup-run-td.sh`, `scripts/setup-run-ra-remastered.sh`)
symlink the Steam install's `CNCDATA/{TIBERIAN_DAWN,RED_ALERT}/CD1/` files into
`build/run-{td,ra-remastered}/`.

`REDALERT/MIXFILE.H` and `TIBERIANDAWN/MIXFILE.H` use `int32_t` with
`__attribute__((packed))` for `SubBlock` and `FileHeader` to match the on-disk
Win32 layout (LP64 `long` is 8 bytes on Linux, 4 bytes on Win32).

## Headless Testing with Xvfb

CI runs the game under a virtual framebuffer:

```bash
Xvfb :99 -screen 0 640x480x24 &
TD_AUTOSTART=1 TD_CHEAT=1 DISPLAY=:99 ./build/cmake-td/td
```

`TD_AUTOSTART=1` bypasses the main menu (set in `TIBERIANDAWN/INIT.CPP`).
`TD_CHEAT=1` sets `Debug_Flag=true` and injects credits, tech unlock, map
reveal, and mission win at fixed frame counts.

The frame-count gate (`[TD] Main_Loop frame N`) in `TIBERIANDAWN/CONQUER.CPP`
is the smoke-test milestone. `scripts/run-td-cheat.sh` verifies all four cheat
milestones pass.
